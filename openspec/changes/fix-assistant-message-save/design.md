## 上下文

当前 `chat.py` 使用 FastAPI 的依赖注入 `get_db()` 获取数据库会话。问题在于：

```
请求进入 → get_db() 创建会话 → chat() 执行 → 返回 StreamingResponse
                                                            │
                                                            ▼
                                         FastAPI 认为请求完成，关闭数据库会话
                                                            │
                                                            ▼
                                         浏览器接收 SSE → stream_chat_response() 执行
                                                            │
                                                            ▼
                                         尝试保存 assistant 消息 ✗ (会话已关闭)
```

## 目标 / 非目标

**目标：**
- 确保 assistant 消息在流式响应期间正确保存到数据库
- 保持用户消息和 assistant 消息的事务一致性

**非目标：**
- 不改变现有的 API 接口
- 不修改前端代码

## 决策

### 决策 1：使用独立的数据库会话管理

**选择方案：** 在流式生成器内部创建独立的数据库会话

**理由：** FastAPI 的依赖注入在 `StreamingResponse` 返回时结束请求上下文。使用 `async_session_maker` 直接在生成器内部管理会话，可以确保会话在整个流式响应期间保持活跃。

**替代方案：**
- 使用 BackgroundTask：复杂且不适合流式场景
- 修改 get_db() 的生命周期：影响全局行为，风险较大

### 决策 2：用户消息与 assistant 消息分事务处理

**选择方案：** 用户消息使用当前会话保存，assistant 消息使用独立会话保存

**理由：** 用户消息需要在流式响应开始前确认保存成功，assistant 消息在流式结束时保存。分离事务可以：
1. 用户消息失败时立即返回错误
2. assistant 消息保存失败不影响已返回的内容

## 风险 / 权衡

| 风险 | 缓解措施 |
|------|----------|
| 独立会话可能导致用户消息与 assistant 消息不在同一事务 | 可接受：消息按时间顺序保存，即使分开也是一致的 |
| 如果流式过程中服务崩溃，用户消息已保存但无回复 | 前端已处理流式中断的情况，用户体验可接受 |
