## 上下文

当前系统在处理工具调用时存在消息存储顺序问题。根据 OpenAI API 规范，`tool` 角色的消息必须紧跟在包含 `tool_calls` 的 `assistant` 消息之后。

当前代码流程：
```
LLM 返回 tool_calls → 执行工具 → 存储工具消息 → 最后存储 assistant 消息
```

这导致数据库中消息顺序为：`user → tool → tool → tool → assistant`，而正确顺序应为：`user → assistant(tool_calls) → tool → tool → tool → assistant(response)`

## 目标 / 非目标

**目标：**
- 修复消息存储顺序，确保符合 OpenAI API 规范
- 解决继续对话时的 400 错误
- 解决前端消息重复显示问题

**非目标：**
- 不修改数据库 schema
- 不修改 API 接口
- 不处理历史数据迁移（已存储的错误顺序数据）

## 决策

### 决策 1：修改消息存储顺序

**选择方案：** 按正确顺序存储消息：
1. **先存储** 包含 `tool_calls` 的 `assistant` 消息（当前代码没有存储这条消息！）
2. **再存储** 每个 `tool` 角色的消息（工具执行结果）
3. **最后存储** 最终的 `assistant` 响应消息

**重要发现：** 当前代码中，包含 `tool_calls` 的 `assistant` 消息只存在于内存中，从未存储到数据库。这是根本原因。

**理由：**
- 消息顺序符合 OpenAI API 规范
- 解决继续对话时的 400 错误
- 历史会话可以正确恢复

**替代方案：**
- 方案 A：先存储临时 assistant 消息 → 复杂，需要后续更新
- 方案 B：重构整个消息存储逻辑 → 改动过大，风险高

### 决策 2：前端过滤 tool 消息

**选择方案：** 在 MessageList 组件中过滤 `role="tool"` 的消息

**理由：**
- 工具调用信息已通过 `assistant.tool_calls` 渲染
- `tool` 消息仅用于构建 LLM 上下文，不应直接显示给用户
- 改动最小，影响范围可控

## 风险 / 权衡

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 已存储的错误顺序数据 | 现有会话可能仍有问题 | 用户可新建会话，或后续添加数据迁移脚本 |
| 前端过滤可能遗漏其他场景 | 某些特殊情况下 tool 消息可能需要显示 | 当前场景下 tool 消息仅为工具结果，不需要单独显示 |

## 迁移计划

无需迁移。此修复仅影响新创建的消息，已存储的消息保持不变。如需修复历史数据，可后续添加迁移脚本。
