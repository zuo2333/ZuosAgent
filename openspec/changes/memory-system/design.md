# 记忆系统技术设计

## 上下文

### 背景

当前项目是一个 LLM 对话服务平台，具备完整的 Agent 能力（工具调用、流式响应、会话管理）。每次对话都是独立的，缺乏跨会话的记忆能力。

### 当前架构

```
用户消息 → Agent 循环 → LLM → 工具调用 → 响应
                ↓
           会话存储 (PostgreSQL)
```

### 约束

- 使用现有的 PostgreSQL 数据库，不引入新的数据存储
- 保持与现有 Agent 循环的兼容性
- 记忆注入不能显著增加响应延迟（<500ms）

## 目标 / 非目标

**目标：**

1. 实现三层记忆架构：短期记忆、长期记忆、用户画像
2. 支持记忆的自动提取、存储、检索
3. 将相关记忆注入 LLM 上下文，实现个性化回复
4. 支持记忆的时间衰减和重要性排序

**非目标：**

1. 不实现多用户系统（当前为单用户场景）
2. 不实现记忆的显式管理界面（仅 API 级别）
3. 不实现复杂的知识图谱推理（仅存储和检索）

## 决策

### 决策 1：向量数据库选型

**选择：pgvector**

**考虑方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| pgvector | 无需新组件，与 PostgreSQL 集成 | 性能一般，功能有限 |
| Qdrant | 高性能，过滤查询强 | 需要独立部署 |
| Chroma | 轻量，Python 友好 | 功能有限 |

**理由：**
- 项目已使用 PostgreSQL，pgvector 无需额外运维
- 记忆规模预期 <10 万条，pgvector 性能足够
- 后期如需迁移，pgvector 与 Qdrant 的向量格式兼容

### 决策 3：长期记忆提取策略

**选择：会话结束时 + 定期检查点**

**考虑方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 实时提取 | 记忆即时可用 | 增加 LLM 调用延迟 |
| 会话结束时 | 简单直接 | 长会话可能丢失中间信息 |
| 会话结束 + 检查点 | 兼顾延迟和完整性 | 实现稍复杂 |

**触发时机：**

```
┌─────────────────────────────────────────────────────────────────┐
│                    长期记忆提取触发条件                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 会话结束                                                     │
│     ├── 用户明确关闭会话                                         │
│     ├── 用户切换到另一个会话（闲置当前会话）                       │
│     └── 用户闲置超过 10 分钟                                      │
│                                                                 │
│  2. 定期检查点（长会话保护）                                      │
│     └── 会话消息数达到 20/40/60... 条时触发                       │
│                                                                 │
│  3. 手动触发                                                     │
│     └── 用户调用 POST /api/memory/extract                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**理由：**
- 会话结束时提取是行业标准做法（ChatGPT Memory、Claude Projects）
- 检查点机制保护长会话不丢失中间记忆
- 不影响正常的 LLM 响应延迟
- 用户闲置检测可通过前端心跳或后端会话超时实现

### 决策 4：记忆注入方式

**选择：按需注入 + 意图检测**

**考虑方案：**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 每次都注入 | 简单，信息完整 | Token 消耗高，简单对话也检索 |
| 按需注入 | 节省 Token，精准 | 需要意图检测逻辑 |
| 完全不注入 | 最省 Token | 无法个性化 |

**理由：**
- 业界标准做法是按需注入（ChatGPT Memory、Claude Projects）
- 简单问候语不需要触发记忆检索
- 用户明确询问历史时才检索相关记忆

**意图检测流程：**

```
┌─────────────────────────────────────────────────────────────────┐
│                      记忆注入意图检测                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   用户输入                                                       │
│      │                                                          │
│      ▼                                                          │
│   ┌─────────────────────────────────────────┐                  │
│   │           意图分类器（轻量级）             │                  │
│   │   规则匹配 + 关键词检测（无 LLM 调用）     │                  │
│   └─────────────────────────────────────────┘                  │
│      │                                                          │
│      ├────────────────────┬────────────────────┐               │
│      ▼                    ▼                    ▼               │
│  [无需记忆]           [需要短期记忆]       [需要长期记忆]        │
│      │                    │                    │               │
│      │                    ▼                    ▼               │
│      │            检索短期记忆            向量检索长期记忆       │
│      │            (摘要+实体)             (top_k 相关记忆)      │
│      │                    │                    │               │
│      └────────────────────┴────────────────────┘               │
│                           │                                     │
│                           ▼                                     │
│                    组装记忆上下文                                 │
│                    注入 System Prompt                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**意图分类规则（示例）：**

| 意图类别 | 触发条件 | 示例输入 |
|----------|----------|----------|
| 无需记忆 | 问候语、简单问答 | "你好"、"什么是 Python" |
| 需要短期记忆 | 引用当前会话、任务延续 | "我们刚才讨论了什么"、"继续上一步" |
| 需要长期记忆 | 引用历史、个人信息、偏好 | "上次那个项目"、"我喜欢什么" |

**规则检测示例：**

### 决策 5：短期记忆压缩策略

**选择：滑动窗口 + LLM 摘要 + 批量实体提取**

**参数：**

```python
COMPRESSION_THRESHOLD = 3000  # Token 阈值
KEEP_RECENT_MESSAGES = 6     # 保留最近 N 条消息
```

**流程：**

```
消息历史 → 检查 Token 数 → 超过阈值？
                              ↓ 是
               保留最近 N 条 + 对早期消息生成摘要 + 提取实体
                              ↓ 否
                        保持原样
```

**实体提取策略：批量提取（压缩时）**

| 选项 | 描述 | 选择 |
|------|------|------|
| 每轮后增量 | 每轮对话后提取新实体 | ❌ 增加调用次数 |
| 压缩时批量 | 只在压缩摘要时一起提取 | ✅ 减少 LLM 调用 |

**理由：**
- 实体信息通常在摘要中也会被提及，无需单独提取
- 减少每轮对话的 LLM 调用次数
- 压缩时的 LLM 调用可以一次性完成：摘要生成 + 实体提取 + 任务状态更新

**一次性 LLM 调用 Prompt：**

```
分析以下对话，输出：
1. 摘要：压缩主要内容和结论（2-3 句话）
2. 实体：提取人物、项目、技术等关键实体
3. 任务：识别待办事项和当前目标

对话内容：
[早期消息列表]
```

## 风险 / 权衡

### 风险 1：记忆质量不可控

**风险**：LLM 提取的记忆可能不准确或遗漏重要信息

**缓解措施**：
- 使用高置信度阈值（>0.7）才存储
- 用户可以通过 API 查看/删除记忆
- 后续可增加用户反馈机制（点赞/删除）

### 风险 2：Token 消耗增加

**风险**：记忆注入会增加 LLM 输入 Token 数

**缓解措施**：
- 设置 Token 预算上限（默认 1000 tokens）
- 智能选择最相关的记忆
- 允许用户关闭记忆功能

### 风险 3：向量检索延迟

**风险**：向量相似度搜索可能增加响应延迟

**缓解措施**：
- 使用 HNSW 索引加速检索
- 限制检索数量（top_k=10）
- 可预加载常用记忆到缓存

### 风险 4：存储成本增长

**风险**：长期记忆持续增长导致存储压力

**缓解措施**：
- 实现遗忘曲线机制，自动降低不重要记忆的权重
- 设置记忆上限，定期清理低重要性记忆
- 支持用户手动清理
